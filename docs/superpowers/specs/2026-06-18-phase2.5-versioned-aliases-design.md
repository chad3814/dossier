# Phase 2.5 — versioned aliases — design

**Date:** 2026-06-18
**Status:** Approved (design)
**Component:** `dossier`; DCC data in `casefiles/dcc/`
**Builds on:** `2026-06-17-phase2-versioned-descriptions-design.md`

## Background

Phase 2 versioned **descriptions** so a gated view shows spoiler-bounded prose. But
**aliases** are still un-versioned — carried as the full registry set at introduction — so a
`--through B1·C1` view leaks alias spoilers (e.g. Carl already shows `Crawler #4,122`,
`Level 34`). Phase 2.5 versions aliases the same way, folded into the **same re-describe
pass** so one run produces both `descriptions.json` and `aliases.json`.

## Decisions

- **Capture in the re-describe pass.** The chapter agent — already reading each chapter —
  additionally reports which of an entity's *not-yet-seen known aliases* appear in this
  chapter's text. One unified pass, no extra reads.
- **Match-only alias list.** The agent is given the entity's known aliases purely to match
  against the chapter text; the prompt forbids using them to inform the description, so
  description spoiler-bounding is unchanged. (Accepted mild exposure: the agent sees
  future-revealing forms in the candidate list, but its output only includes forms present
  in the current chapter.)

## Data model

```ts
/** An alias that first becomes known at `anchor` (Phase 2.5). */
interface AliasEvent {
  id: string;
  anchor: string;
  alias: string;
}
```

Persisted at `casefiles/dcc/log/aliases.json` — an array sorted by `anchorSortKey` then id,
mirroring `descriptions.json`. Separate stream from appearances and descriptions.

## `materialize` overlay (unified, leak-safe)

`materialize` gains `aliases?: AliasEvent[]`. Both the description and alias overlays now use
one rule, keyed on **whether an entity was processed** (has ≥1 event in the stream):

- **Processed entity** → value derived from that entity's events with `anchor ≤ cutoff`
  *only* (no blob). Descriptions: the latest such version (empty string if none ≤ cutoff).
  Aliases: `cleanAliases(canonicalName, [aliases of events ≤ cutoff])` (empty if none).
- **Unprocessed entity** (no events in the stream — e.g. a partial/pilot run hasn't reached
  it) → keep the appearance-log blob value.

This closes a latent leak: an entity that appears early but whose first description/alias
lands in a later chapter no longer falls back to the spoilery full blob — it shows only what
is known by the cutoff. (Refines the Phase-2 description overlay, which keyed on "no event ≤
cutoff → blob"; the Phase-2 full run had not yet run, so there is no migration impact.)

## Re-describe pass changes

- **Workflow (`workflow/redescribe-book.js`):** per chapter, for each appearing entity the
  agent now also receives `candidateAliases` (the entity's registry aliases not yet emitted,
  tracked in workflow state) and returns `aliasesSeenHere` — the subset whose exact form
  appears in this chapter. Each becomes an `AliasEvent` at the chapter's anchor; emitted
  aliases are marked so later chapters only emit *first* occurrences. The workflow now
  returns `{ bookNumber, descriptionEvents, aliasEvents }`.
- **`gen-redescribe.ts`:** injects an `aliasesById` map (`id → registry aliases`) so the
  workflow can compute per-chapter candidates.
- **Reconciliation (completeness):** after the full pass, any registry alias never matched in
  text gets an `AliasEvent` at the entity's **last** appearance anchor — preserved at the
  full-series view, never leaked early. Guarantees the full-series alias set equals the
  registry's. Implemented as a deterministic post-step over the pass output + the log.

## Outputs & integration

- `view` loads `aliases.json` (if present) and passes it to `materialize` alongside
  `descriptions.json`.
- `validate-descriptions` gains a sibling `validate-aliases` check (or a shared validator):
  per entity, alias events in non-decreasing anchor order; every event's id known; after a
  full run, every registry alias is represented.
- The Phase-1 reproduction test (`test/migration-repro.test.ts`) now also excludes `aliases`
  from the deep-equal (versioned/regenerated, like description/significance). `id`,
  `appearances`, `tags`, `type`, `canonicalName`, `firstAppearance` still reproduce exactly.

## Testing

- **Unit:** the unified overlay — processed entity shows only ≤-cutoff aliases/descriptions
  (empty when none), unprocessed keeps blob; alias set at successive cutoffs grows correctly;
  reconciliation places unplaced aliases at the entity's last appearance.
- **Agent output:** validated in the re-run pilot (alias events bounded to their chapter;
  spot-check that early forms like `Crawler #4,122` first appear at the right chapter).

## Then

Re-run the re-describe pass over all 8 books once, producing both `descriptions.json` and
`aliases.json`; reconcile; validate; commit + push.
